## 3. 阶段一：M0 基础设施完善

### 步骤 1：数据库 DDL 脚本生成 ✅

**状态**：**已完成**（2026-06-15）

**目标**：创建 MySQL 建表 SQL 文件

**产出文件**：`damai-parent/sql/ddl.sql`

**实际产出**：13 张表 DDL，按业务域分组（utf8mb4 / InnoDB），含完整字段注释、索引、状态码对齐说明：
- 分片表 3 张（`t_order` / `t_order_detail` / `t_pay`，分片键 `user_id`，与 `t_order` 同分片组）
- 非分片表 10 张（用户/节目/基础数据/本地消息表）
- 状态码与 Java 枚举严格对齐：`OrderStatusEnum`(1-5) / `PayStatusEnum`(1-5) / `ProgramStatusEnum`(0-3) / `UserStatusEnum`(-1/0/1)
- 票档表 `t_ticket_type` 含 `sale_stock` / `locked_stock`，支撑 L3 级 DB 条件更新防超卖
- 本地消息表 `t_local_message` 含状态机（0待发送/1已发送/2已确认/3失败）与重试退避字段

**产出表清单**：

| 业务域 | 表名 | 说明 | 分片策略 |
|---|---|---|---|
| 用户域 | `t_user` | 用户表（手机号、密码哈希、昵称、头像、状态等） | 不分片，单库 |
| 用户域 | `t_viewer` | 观演人表（姓名、证件类型、证件号，关联 user_id） | 不分片，单库 |
| 节目域 | `t_program` | 节目表（标题、艺人、场馆、城市、类型、状态、热度等） | 不分片，单库 |
| 节目域 | `t_show` | 场次表（关联 program_id、演出时间、开售时间等） | 不分片，单库 |
| 节目域 | `t_ticket_type` | 票档表（关联 show_id、票档名称、价格、库存等） | 不分片，单库 |
| 订单域 | `t_order` | 订单表（order_id 雪花算法、user_id 分片键、状态、金额等） | 4 库 × 8 表 = 32 分片，分片键 `user_id` |
| 订单域 | `t_order_detail` | 订单明细表（关联 order_id、场次、票档、数量等） | 与 `t_order` 同分片组 |
| 支付域 | `t_pay` | 支付单表（关联 order_id、支付方式、状态、金额等） | 与 `t_order` 同分片组 |

| 基础数据 | `t_region` | 地区表 | 不分片，单库 |
| 基础数据 | `t_banner` | 轮播图表 | 不分片，单库 |
| 基础数据 | `t_dict` | 字典表 | 不分片，单库 |
| 基础设施 | `t_local_message` | 本地消息表（事务消息，保证最终一致） | 不分片，单库 |

**实际落地要点**：
- 所有表统一含 `id`（BIGINT）、`create_time`、`update_time`、`is_deleted` 通用字段（配合 service-starter MyBatis-Plus 自动填充与全局逻辑删除）

- 索引设计已落地：`t_user.mobile` 唯一索引、`t_order(user_id, status)` 联合索引、`t_viewer(user_id, id_card_type, id_card_no)` 唯一约束等
- 证件号、密码等敏感字段注释已标注加密存储 / BCrypt 哈希
- 待补：初始测试数据 `damai-parent/sql/dml.sql`（将在步骤 31 联调收尾时生成）

| 字段 | ES 类型 | 分类 | 说明 |

| `programId` | keyword | 主键 | 节目 ID，精确匹配 |

| `venue` | text + keyword 子字段 | 搜索 | 场馆，ik 分词 |
| `description` | text | 搜索 | 节目介绍，辅助全文检索 |

| `cityCode` | keyword | 筛选 | 城市编码（关联 `t_region.code`） |

| `status` | keyword | 筛选 | 节目状态（ProgramStatusEnum） |
| `priceMin` | double | 范围 | 最低票价 |
| `priceMax` | double | 范围 | 最高票价 |
| `showStart` | date | 范围 | 演出开始时间（首场） |
| `showEnd` | date | 范围 | 演出结束时间（末场） |
| `saleStart` | date | 范围 | 开售时间（抢票起跑点） |
| `heat` | integer | 排序 | 热度值 |
| `posterUrl` | keyword（index=false） | 辅助 | 海报 URL，仅存储不索引 |
| `createTime` / `updateTime` | date | 系统 | 创建/更新时间 |

**实际落地要点**：
- 索引名：`damai_program`，3 主分片 + 1 副本，刷新间隔 1s
- 分词器：`ik_max_word`（索引时细粒度）+ `ik_smart`（查询时智能），需预装 IK 插件
- `dynamic: strict` 拒绝未知字段，防脏数据污染索引
- README 含多条件分类浏览（BoolQuery）与智能搜索（multi_match + highlight）查询示例，供后续 ES Manager 实现参考
- **字段名与原计划差异**：实际采用 `typeCode`/`cityCode`（与 DDL 表字段、外键语义一致），优于原计划的 `type`/`city`；并新增 `description`/`posterUrl`/`showEnd`/`saleStart` 等字段以覆盖完整搜索场景

### 步骤 3：`damai-service-starter` 公共自动装配模块生成 ✅

**状态**：**已完成**（2026-06-15）—— 全部 9 个子包（3.1-3.9）均已生成

**目标**：所有业务服务依赖的公共 Starter，封装中间件自动配置，避免每个服务重复配置

**文件位置**：`damai-parent/damai-service-starter/`

**产出内容**（按子包）：

#### 3.1 模块骨架 ✅
- ✅ `pom.xml`（依赖 common + Spring Boot AutoConfiguration + Redisson + Kafka + ES + MyBatis-Plus + Sentinel + Seata，所有中间件依赖 `optional=true`）
- ✅ `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（已注册 3.1-3.5 的 5 个 AutoConfiguration 类）

#### 3.2 `redis/` 子包 ✅
- ✅ `RedissonAutoConfiguration.java`：Redisson 自动配置（`@AutoConfiguration(after = RedissonAutoConfigurationV2.class)`，注册 BloomFilterHelper/DistributedLockHelper Bean）
- ✅ `BloomFilterHelper.java`：布隆过滤器工具（`tryInit` 幂等初始化 / `add` 写入 / `mightContain` 判断，对应 `RedisKeyConstant.BLOOM_PROGRAM_EXISTS` / `BLOOM_USER_MOBILE`）
- ✅ `DistributedLockHelper.java`：分布式锁工具（`tryLock` 无等待 / `tryLockWithTimeout` 带超时 / `executeWithLock` 带业务回退模板方法，看门狗自动续期，`isHeldByCurrentThread` 防误释放）

#### 3.3 `kafka/` 子包 ✅
- ✅ `KafkaProducerAutoConfiguration.java`：Kafka 自动配置（`@ConditionalOnClass(KafkaTemplate)`）
- ✅ `KafkaMessageHelper.java`：消息发送工具（`sendSync` 同步阻塞等待 Broker 确认 / `sendAsync` 异步回调 / `sendAsyncWithTrace` 含 traceId 注入，统一 JSON 序列化，失败抛 `SystemException(KAFKA_SEND_ERROR)`）
- ✅ `BaseKafkaConsumer.java`：通用消费者基类（模板方法模式，手动 ACK + traceId Header 透传 + JSON 反序列化 + 异常重试 → DLQ，子类仅需实现 `handleMessage`）

#### 3.4 `sentinel/` 子包 ✅
- ✅ `SentinelAutoConfiguration.java`：Sentinel 自动配置（`@ConditionalOnClass(SphU)`，注册 HotParamLimitConfig Bean）
- ✅ `HotParamLimitConfig.java`：热点参数限流规则（`createOrder` 按 programId 500 QPS / `programDetail` 按 programId 2000 QPS / `stockDeduct` 按 ticketTypeId 1000 QPS，PostConstruct 加载）

#### 3.5 `es/` 子包 ✅
- ✅ `ElasticSearchAutoConfiguration.java`：ES 客户端自动配置（`@ConditionalOnProperty(spring.elasticsearch.uris)`，注册 EsQueryHelper Bean）
- ✅ `EsQueryHelper.java`：通用查询封装（`search(BoolQuery)` 多条件筛选 / `search(Query)` 自定义查询 / `getById` 精确查询 / `index` 新增更新 / `deleteById` 删除 / `boolBuilder` 工厂方法，基于 ES 8.x Java API Client）
- ✅ `EsSearchResult.java`：搜索结果封装（total/pageNum/pageSize/list，含 getTotalPages/hasNext）

#### 3.6 `mybatis/` 子包 ✅
- ✅ `MybatisPlusAutoConfiguration.java`：MyBatis-Plus 自动配置（注册 `MybatisPlusInterceptor` 分页插件 MySQL 方言 + `MetaObjectHandler` 自动填充 Bean）
- ✅ `CreateUpdateMetaHandler.java`：自动填充处理器（insert 填充 create_time/update_time，update 刷新 update_time，仅填充 null 值不覆盖业务设置值）
- ℹ️ ShardingSphere 分片规则不在此统一配置（各服务分片规则不同：order 按分片、user/program 不分片），由各服务 config 包自行定义

#### 3.7 `seata/` 子包 ✅
- ✅ `SeataAutoConfiguration.java`：Seata AT 自动配置（`@ConditionalOnClass(GlobalTransactional)` + `@ConditionalOnProperty(seata.enabled)` 条件装配，AT 数据源代理与全局事务扫描由 spring-cloud-starter-alibaba-seata 自动完成）
- ℹ️ 各业务服务在需分布式事务的方法上标注 `@GlobalTransactional` 即可，遵循技术文档 §8.3.2 长事务规避策略（仅关键 DB 写纳入事务，Kafka/出票/通知移出事务外）

#### 3.8 `web/` 子包 ✅
- ✅ `GlobalExceptionHandler.java`：全局异常处理器 `@RestControllerAdvice`（区分 BizException→DEBUG 日志 / SystemException→ERROR 日志 + 堆栈 / 参数校验异常→PARAM_INVALID / 兜底 Exception→SYSTEM_ERROR，统一注入 traceId）
- ✅ `TraceIdFilter.java`：traceId 拦截器（Servlet Filter 实现，`@Order(HIGHEST_PRECEDENCE)` 最高优先级，从 Header 提取 traceId → 绑定 ThreadLocal + MDC → 响应头回写 → finally 清理防线程池串号）
- ✅ `CorsAutoConfiguration.java`：跨域自动配置（`CorsFilter` Bean，`allowedOriginPattern("*")` + 凭证 + 预检缓存 3600s，暴露 X-Trace-Id/X-User-Id 响应头）

#### 3.9 `idempotent/` 子包 ✅
- ✅ `IdempotentToken.java`：幂等令牌自定义注解（`@Target(METHOD)`，`paramName` 默认 "token"）
- ✅ `PreventDuplicate.java`：防重锁自定义注解（`prefix` 业务前缀 + `expireSeconds` TTL 默认 5s）
- ✅ `IdempotentTokenAspect.java`：幂等令牌 AOP 切面（`@Order(HIGHEST_PRECEDENCE+1)` 先执行，从请求参数/Header 提取 token，Redis Lua GETDEL 原子消费，nil=重复提交抛 ORDER_IDEMPOTENT_TOKEN_INVALID）
- ✅ `PreventDuplicateAspect.java`：防重锁 AOP 切面（`@Order(HIGHEST_PRECEDENCE+2)` 后执行，从 Header 提取 userId 构造 `lock:submit:{userId}:{prefix}` Key，Redisson trySet 短窗口锁，业务完成后主动释放）

**同步操作**：确认父 pom.xml 中 `damai-service-starter` 的 `<modules>` 声明已启用

### 步骤 13：`damai-program` 库存预热与 Lua 预扣 ⭐（核心）

### 步骤 20：`damai-order` 抢票下单核心 Service ⭐（最核心）

**文件位置**：`com.damai.order.service/`、`com.damai.order.service.impl/`

**产出内容**：

| 文件 | 方法 | 说明 |
|---|---|---|
| `OrderService.java` | — | 接口定义 |
| `OrderServiceImpl.java` | `generateToken(userId)` | 生成幂等令牌：UUID → Redis `idempotent:token:{token} = userId`，TTL 10min |
| | **`createOrder(userId, request)`** | **⭐核心下单全链路** |
| | | 1. 幂等校验：Redis GETDEL 原子消费令牌（令牌无效 → 拒绝重复提交） |
| | | 2. 防重锁：Redis 短窗口 5s 分布式锁（防止用户狂点） |
| | | 3. 库存预扣：Feign 调用 program 服务 `StockDeductService` 执行 Redis Lua 预扣（库存不足 → 快速失败返回） |
| | | 4. 落订单：DB 写入待支付订单（Seata AT 包裹关键写操作） |
| | | 5. 本地消息表：与订单同一事务写入消息记录 |
| | | 6. 延迟消息：Kafka 投递超时取消延迟消息（15min 级别） |
| | | 7. 异步通知：Kafka 投递"订单已创建"消息（通知 program 服务异步扣 DB 库存） |
| | | 8. 返回：订单号 + 待支付金额 + 支付截止时间（下单后 15 分钟） |
| | `cancelOrder(userId, orderId)` | 主动取消（校验状态=待支付 → 归还 Redis 库存 Feign → 更新订单状态 → 异步归还 DB 库存 Kafka） |
| | `queryOrders(userId, status, pageNum, pageSize)` | 按状态分页查询（命中分片，高效） |
| | `queryOrderDetail(userId, orderId)` | 订单详情 |
| `IdempotentTokenService.java` | `generateToken()` / `consumeToken()` | 幂等令牌生成与消费（Redis GETDEL 原子操作） |

**核心链路时序**（呼应 PRD §8.1）：

```
用户 → 网关(限流) → order(领令牌) → order(提交下单)
   → 幂等校验(消费令牌) → 防重锁(5s) → Lua预扣Redis(Feign)
   → 落单DB(Seata) → 本地消息表 → Kafka延迟消息(15min)
   → Kafka异步通知DB库存扣减 → 返回订单号+金额+截止时间
```

### 步骤 21：`damai-order` 延迟消息生产与消费

**文件位置**：`com.damai.order.mq/`

**产出内容**：

|---|---|
| `OrderCreatedProducer.java` | 下单成功后投递延迟消息到 Kafka（多级延迟方案：投递到 `damai-order-cancel-delay` 15min 级别 Topic） |

- 失败重试 → 死信队列（DLQ）人工介入

**目标**：Spring Cloud Gateway 统一入口

**文件位置**：`damai-parent/damai-gateway/`

**产出内容**：

#### 29.1 模块骨架
- `pom.xml`（依赖 common + Spring Cloud Gateway Starter + Sentinel + Nacos Discovery）
- `GatewayApplication.java`：启动类
- `bootstrap.yml`：服务名 `damai-gateway`、端口 `8080`、Nacos 注册发现

#### 29.2 过滤器（`filter/`）
- `JwtAuthGlobalFilter.java`：**JWT 鉴权过滤器**（全局过滤器）
  - 从请求头/Authorization 提取 JWT
  - 校验签名与过期时间
  - 将 `userId`、角色注入请求头透传至下游服务
  - 白名单放行（登录、注册、公开节目浏览/搜索等接口）

## 11. 附录：四层分层架构约束（每步必须遵守）

| 层 | 职责 | 可调用 | 禁止 |
|---|---|---|---|
| **Controller** | 参数校验、调 Service、组装返回 | Service | 直接调 Manager/DAO；含业务逻辑 |
| **Service** | 业务编排、事务边界、调 Manager/Feign | Manager、Feign Client、同模块 Service | 直接调 DAO；跨层 |
| **Manager** | 缓存/中间件/组合 DAO 的胶水层 | DAO、Redis、ES、Kafka | 被 Controller 直接调用 |
| **DAO** | 数据访问（MyBatis-Plus Mapper） | DB | 含业务逻辑；调 Service |

- `@Transactional` 只能在 **Service 层**
- 方法尽量不超过 **80 行**，职责单一
- Redis/ES/Kafka 操作**必须**通过 Manager 层封装，Service 不直接操作中间件
