## 2. 总体架构

### 2.1 架构分层总览

系统采用**微服务 + 网关 + 中间件集群**的经典分层，并贯彻 **CQRS（读写分离）** 思想：写流量走 MySQL/Redis，读流量走 ElasticSearch/缓存，二者通过异步消息保持最终一致。

```
                       ┌─────────────────────────────────┐
                       │          客户端 (App/H5)          │
                       └────────────────┬────────────────┘
                                        │ HTTPS
                       ┌────────────────▼────────────────┐
                       │     Spring Cloud Gateway        │  路由 / JWT鉴权 / 全局限流 / 跨域
                       └────────────────┬────────────────┘
                                        │
        ┌───────────┬───────────┬───────┴────────┬────────────┐
        ▼           ▼           ▼                ▼            ▼
   用户服务    节目服务     订单服务          支付服务      基础数据服务
  (user)     (program)    (order)           (pay)         (base)
        │           │           │                │            │
        │     ┌─────┴─────┐     │                │            │
        │     ▼           ▼     │                │            │
        │   写:MySQL  读:ES     │                │            │
        │                       │                │            │
        └───────────┬───────────┴────────────────┘            │
                    │  Feign(同步) / Kafka(异步)               │
        ┌───────────▼─────────────────────────────────────────▼───┐
        │                    基础设施层                            │
        │  Nacos(注册+配置)  Redis(Redisson)  Kafka  Seata(AT)    │
        │  ElasticSearch  MySQL(ShardingSphere)  Sentinel          │
        └──────────────────────────────────────────────────────────┘
```

### 2.2 读写职责划分（CQRS）
| 数据流向 | 承载组件 | 说明 |
|---|---|---|
| 写（下单/支付/库存变更） | MySQL（分片） + Redis（预扣） | 强一致/事务保障 |
| 读（浏览/搜索/详情） | ElasticSearch + Redis 缓存 | 高吞吐、多维筛选 |
| 同步 | Kafka 异步消息 + 定时校对任务 | ES 与 MySQL 最终一致 |

### 3.2 ⚠️ 已知集成风险与兜底
**风险**：`Spring Boot 4.0 + Spring Cloud 2025.1 + Nacos` 组合存在社区在跟踪的集成问题（[GitHub Issue #4183](https://github.com/alibaba/spring-cloud-alibaba/issues/4183)），可能出现配置加载/服务发现相关的兼容性摩擦。

**兜底策略**（文档强制写入「风险清单」）：
1. 锁定 `spring-cloud-alibaba 2025.1.0.0` 的**精确小版本**，不使用 SNAPSHOT。
2. 持续跟踪 #4183 的官方修复，及时打补丁版本。
3. 配置中心客户端启用**本地快照兜底**（Nacos 客户端默认支持），在 Nacos 不可达时使用上次成功快照，保证服务可启动。
4. 关键启动配置（DB 连接最小集、Nacos 地址）保留在 `bootstrap.yml`，避免完全依赖远程配置导致冷启动失败。

### 3.3 关键命名空间注意
Spring Boot 4.0 全面采用 **Jakarta EE** 命名空间（`jakarta.*`），所有 `javax.servlet.*`、`javax.validation.*` 等需替换为 `jakarta.*`。代码与依赖需严格检查。

## 5. 微服务拆分与边界

### 5.1 服务职责
| 服务 | 端口建议 | 职责 | 是否核心 |
|---|---|---|:---:|
| damai-gateway | 8080 | 统一入口、JWT 鉴权、限流、路由 | ✅ |
| damai-user | 8081 | 注册/登录/鉴权/个人信息/观演人 | ✅ |
| damai-program | 8082 | 节目浏览/搜索/详情/**库存预热与预扣** | ✅ |
| damai-order | 8083 | 下单/查询/取消/**超时取消消费** | ✅ |
| damai-pay | 8084 | 支付/回调/退款/对账 | ✅ |
| damai-base | 8085 | 节目类型/地区/轮播/字典 | 支持 |

> **库存归属说明**：库存（预热、预扣、归还）逻辑放在 **program 服务**，因为它最了解"场次+票档"的语义；order 服务通过 Feign/Kafka 与之协作。这是领域驱动的设计取舍。

### 5.2 服务间通信方式
| 场景 | 方式 | 例子 |
|---|---|---|
| 强依赖、需同步结果 | **OpenFeign**（同步） | order 调 program 做实时库存校验 |
| 解耦、可异步、最终一致 | **Kafka**（异步） | 支付成功通知订单、订单超时取消、DB 库存异步扣减 |
| 跨服务事务 | **Seata AT**（同步全局事务） | 下单扣库存（仅关键写纳入） |

## 6. 四层分层架构约束（约束 §2）

严格遵循经典四层，**禁止跨层调用**：

| 层 | 职责 | 可调用 | 禁止 |
|---|---|---|---|
| **Controller** | 参数校验、调用 Service、组装返回 | Service | 直接调 Manager/DAO；含业务逻辑 |
| **Service** | 业务编排、事务边界、调用 Manager/Feign | Manager、Feign Client、其他 Service（同模块） | 直接调 DAO；跨层 |
| **Manager** | 通用业务封装（缓存、第三方对接、组合 DAO） | DAO、Redis、ES、Kafka | 被 Controller 直接调用 |
| **DAO** | 数据访问（MyBatis-Plus Mapper） | DB | 含业务逻辑；调用 Service |

**约束细则**：
- 一个方法尽量不超过 80 行（约束 §3），职责单一；过长须拆分。
- 事务注解 `@Transactional` 只能在 Service 层。
- Manager 层是缓存/DB/中间件的"胶水层"，Service 不直接操作 Redis/ES。

## 7. 存储设计

### 7.1 MySQL + ShardingSphere 分库分表

#### 7.1.1 分片目标
- 解决**订单表**数据量膨胀（单表 > 千万行性能下降）。
- 均匀打散热点用户的订单数据。

#### 7.1.2 分片策略
| 表 | 分片键 | 分片算法 | 库/表数量（示例） |
|---|---|---|---|
| `t_order` | `user_id` | 取模/一致性哈希 | 4 库 × 8 表 = 32 分片 |
| `t_order_detail` | `order_id`（与 order 同分片组） | 与主表绑定 | 同上 |
| `t_pay` | `order_id` | 与 order 同分片组 | 同上 |
| `t_program` / `t_show` / `t_ticket_type` | 不分片 | 单库 | 1 库 |

> **分片键选择理由**：订单按 `user_id` 查询最高频（"我的订单"），用 `user_id` 分片使该查询精准命中单分片。运营按时间/节目统计则走 ES 或汇总表，避开跨分片聚合。

#### 7.1.3 分片键生成
- `order_id` 采用**雪花算法**，并将 `user_id` 的低位嵌入 `order_id` 末几位，确保 `order_id` 与 `user_id` 落同一分片（避免 join 跨库）。

#### 7.1.4 扩容方案
- 采用**一致性哈希**或**2 的幂扩容**（4→8 库），减少数据迁移量；扩容期双写灰度。

### 7.2 Redis 数据结构规划

| 用途 | Key 规范 | 数据结构 | TTL |
|---|---|---|---|
| 节目详情缓存 | `program:detail:{programId}` | String(JSON) | 30min + 随机抖动 |
| 场次票档库存（预扣） | `stock:{showId}:{ticketTypeId}` | String(int) / Hash | 与开售周期一致 |
| 幂等令牌 | `idempotent:token:{token}` | String(userId) | 10min（一次性消费） |
| 防重锁 | `lock:submit:{userId}:{biz}` | String | 5s |
| 分布式锁（库存） | `lock:stock:{showId}:{ticketTypeId}` | Redisson RLock | 看门狗续期 |
| 布隆过滤器（节目存在性） | `bloom:program:exists` | Redisson RBloomFilter | 持久 |
| 布隆过滤器（手机号唯一） | `bloom:user:mobile` | Redisson RBloomFilter | 持久 |
| 用户 Session | `session:jwt:{userId}` | String | 7d（RefreshToken） |
| 短信验证码 | `sms:code:{mobile}` | String | 5min |
| 首页/频道缓存 | `home:{channel}:{city}` | String(JSON) | 5min |

> **库存 Key 设计要点**：库存预扣必须用 **Lua 脚本**操作（§9.1），保证"读取-判断-扣减"原子性，避免 race condition。

### 7.3 ElasticSearch 索引设计

#### 7.3.1 节目索引（`program`）
| 字段 | 类型 | 说明 |
|---|---|---|
| programId | keyword | 节目 ID（精确匹配） |
| title | text(ik_max_word) | 节目名（中文分词，支持搜索） |
| artist | text(ik_max_word) | 艺人 |
| venue | text / keyword | 场馆 |
| city | keyword | 城市（聚合筛选） |
| type | keyword | 节目类型（演唱会/话剧…） |
| showTime | date | 演出时间（范围筛选/排序） |
| priceMin / priceMax | double | 价格区间 |
| status | keyword | 状态（在售/下架） |
| heat | integer | 热度（排序） |

- **分词器**：`ik_max_word`（索引时细粒度）+ `ik_smart`（查询时）。
- **同步策略**：节目发布/变更 → 发 Kafka → ES 消费者更新索引；**定时校对任务**每小时全量对账，兜底消息丢失。

## 8. 中间件方案

### 8.1 Nacos（注册 + 配置中心，约束 §7）
- **注册中心**：服务实例自动注册与发现。
- **配置中心**：所有环境相关配置托管（DB 连接、Redis 地址、Kafka 地址、限流阈值等），**禁止本地硬编码**。
- **命名空间隔离**：`dev` / `test` / `prod` 三个 namespace。
- **配置分组**：按服务分组，公共配置放 `COMMON_GROUP`，并通过 `extension-configs` 共享。
- **本地快照兜底**：Nacos 客户端默认缓存上次成功配置，断网时可启动（§3.2 兜底）。

### 8.2 Kafka（异步解耦，约束 §4）

#### 8.2.1 Topic 规划
| Topic | 生产者 | 消费者 | 用途 |
|---|---|---|---|
| `order-created` | order | program/库存 | 异步扣 DB 库存 |
| `order-cancel-delay` | order | order（自身，延迟消费） | 订单超时取消 |
| `pay-result` | pay | order | 支付结果通知订单 |
| `program-sync-es` | program | program(ES消费者) | 节目变更同步 ES |

#### 8.2.2 延迟队列实现
Kafka 原生不支持延迟队列，采用**多级延迟 Topic + 轮转**方案：
- 创建若干固定延迟级别的 Topic（如 1m / 5m / 15m / 30m）。
- 消费者按"投递时间 + 延迟级别"判断是否到期：未到期则重投到下一级或原地等待，到期则执行取消。
- **15 分钟订单超时**：投递到 `15m` 级别 Topic，到期消费执行取消 + 归还库存。

> 也可用 `kafka-delay-queue` 思路或 RocketMQ 的延迟等级作为替代，本期以 Kafka 多级方案落地。

#### 8.2.3 消费幂等
- 消费端以"业务唯一键（订单号/支付单号）+ Redis 去重"保证幂等，避免重复消费导致状态错乱。
- 消费失败 → 重试 → 进死信队列（DLQ）人工介入。

### 8.3 Seata AT（分布式事务，约束 §5）

#### 8.3.1 适用场景
- **下单扣库存**（order 写订单 + program 扣库存）跨服务，需强一致时用 Seata AT 兜底。

#### 8.3.2 事务边界与长事务规避（关键）
| 操作 | 是否纳入全局事务 | 原因 |
|---|---|---|
| order 写订单（DB） | ✅ 纳入 | 核心写 |
| program 扣 Redis 库存 | ❌ 不纳入 | Redis 非 Seata RM，且预扣在事务前完成 |
| program 扣 DB 库存 | ✅ 纳入 | 核心写 |
| 发 Kafka（异步通知） | ❌ 不纳入 | IO 耗时，放事务外（事务提交后发送） |
| 出票/通知 | ❌ 不纳入 | 耗时，异步 |

> **设计原则**：Seata 全局事务**只包关键 DB 写**，避免长事务占用全局锁。耗时操作（MQ、出票、第三方调用）一律移出事务边界，通过本地消息表 + 重试保证最终一致。

#### 8.3.3 与本地消息表配合
- 对于"必须成功"的跨服务异步通知，采用**本地消息表**：业务表与消息表在同一本地事务写入，定时扫描消息表投递 Kafka，保证"业务成功 → 消息必达"。

### 8.4 Sentinel（限流熔断）
| 位置 | 规则 | 例子 |
|---|---|---|
| 网关 | 全局 QPS、IP 维度限流 | 单 IP 每秒 ≤ N 次 |
| order 下单接口 | 接口 QPS 限流 + **热点参数限流** | 按 `programId` 限流，防单节目打垮 |
| Feign 调用 | 熔断降级 | program 调用超时 → 降级返回"系统繁忙" |
| 依赖保护 | DB/ES 慢调用熔断 | 防止级联雪崩 |

> **热点参数限流**是抢票场景的关键：避免一个热门节目的流量挤垮整个订单服务，按 `programId` 维度独立限流。

### 8.5 Redisson
- **分布式锁**：`RLock`（看门狗自动续期）——库存极端竞争场景串行化。
- **布隆过滤器**：`RBloomFilter`——节目存在性、手机号唯一性预判。
- **限流器**：`RRateLimiter`——分布式限流补充。

## 9. 高并发抢票核心方案 ⭐（重头戏）

### 9.1 库存预扣 Lua 脚本设计思路

库存预扣必须**原子**完成"读-判-扣"，否则并发下会超卖。核心思路（伪代码级签名，不含完整实现）：

```
-- KEYS[1] = stock:{showId}:{ticketTypeId}
-- ARGV[1] = 扣减数量
-- 返回: 1=成功, -1=库存不足, -2=Key不存在

local stock = redis.call('GET', KEYS[1])
if not stock then return -2 end
if tonumber(stock) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
end
return -1
```

**为什么必须 Lua**：
- 普通 `GET` → `DECRBY` 两步操作之间存在时间窗，并发下多个线程都读到 `stock >= n`，全部扣减导致超卖。
- Lua 脚本在 Redis 中**单线程原子执行**，杜绝竞态。

**调用方式**：通过 Redisson 的 `RScript.eval()` 执行，脚本预加载（`SCRIPT LOAD`）减少传输开销。

### 9.2 三级防超卖
| 层级 | 手段 | 触发条件 | 性能 |
|---|---|---|---|
| L1 | Redis Lua 预扣 | 每次下单 | 极高（核心拦截层） |
| L2 | Redisson 分布式锁 | 仅极端竞争/座位级售卖时 | 中（牺牲并发换强串行） |
| L3 | DB 条件更新 `UPDATE ... SET stock=stock-? WHERE id=? AND stock>=?` | 异步落 DB | 低（最终正确性底线） |

> 实践中 L1 挡住 99.9% 流量；L3 保证即使 Redis 与 DB 短暂不一致，DB 层面也不会超卖。

### 9.3 防缓存三连击

#### 9.3.1 防穿透（查不存在的数据）
- **布隆过滤器**：节目发布写入布隆；查询前先过布隆，不存在的 programId 直接返回，不打 DB。
- 对空结果也缓存短时间（`null` 占位，短 TTL），防止恶意查询。

#### 9.3.2 防击穿（热点 Key 失效）
- 热点节目详情采用**逻辑过期**：
  - 缓存值内含 `logicalExpire` 字段，到期不立即删除。
  - 第一个发现的线程获取**互斥锁**异步重建缓存，其余线程返回旧值。
  - 用户感知无延迟，避免击穿。

#### 9.3.3 防雪崩（大量 Key 同时失效）
- TTL = `base + random(bound)`，避免同时过期。
- 多级缓存（本地 Caffeine + Redis），Redis 失效时本地兜底。
- 关键数据开售前预热。

### 9.4 抢票核心链路时序（呼应 PRD §8.1）
```
下单请求 → 幂等校验 → Lua预扣(Redis) → 落单(DB,分片) → 投延迟消息(Kafka)
   │              │           │              │                │
   │              │           │              │                └─ 异步: DB库存扣减/对账
   │              │           │              └─ Seata AT 关键写(如需强一致)
   │              │           └─ 失败立即返回(库存不足)
   │              └─ 令牌一次性消费(Redis)
   └─ 网关层已做热点参数限流
```
**关键设计**：核心链路只触碰 Redis + 必要 DB 写，MQ 与对账全部后置异步，保证 P99 < 300ms。

## 10. 幂等性与防重设计（PRD §10.5 落地）

### 10.1 下单幂等（令牌机制）
```
1. 前端进入下单页 → 调 /order/token 领取幂等令牌(token=UUID)
2. token 存 Redis: idempotent:token:{token} = userId, TTL=10min
3. 提交下单时携带 token
4. 后端用 Lua 原子消费 token（GETDEL）：返回值存在=首次提交=放行; nil=重复提交=拒绝
5. 业务执行(预扣+落单)
```
> 用 `GETDEL`/Lua 消费保证"判断+删除"原子，避免并发下两个请求都判定为首次。

### 10.2 防重锁（短窗口）
- key = `lock:submit:{userId}:{biz}`，TTL=5s，防止用户狂点。
- 与令牌互补：令牌防"逻辑重复"，防重锁防"物理连点"。

### 10.3 支付回调幂等
- 回调可能被支付方重复投递：以**支付单号**为幂等键，Redis 标记 + 状态机校验双重保障。
- 订单状态机不可逆（已支付 ≠ 退回待支付），即使收到重复/乱序回调也安全。

## 11. 安全设计

### 11.1 认证授权
- **JWT** 无状态认证：网关解析 JWT，校验签名与过期，将 `userId`/角色注入请求头透传。
- **RefreshToken** 存 Redis，支持主动吊销（登出/改密）。
- 细粒度权限：敏感接口（退款、配置）在服务内校验角色。

### 11.2 网关安全
- 统一鉴权过滤器（白名单：登录/注册/公开节目）。
- 接口签名校验（防篡改，可选）。
- 跨域 CORS 配置。

### 11.3 防刷限流
- 网关层：IP/用户/接口多维限流（Sentinel）。
- 业务层：短信验证码 60s 限频、注册防刷。
- 异常流量识别（短时高频访问）→ 临时封禁。

### 11.4 数据安全
- 密码 BCrypt 加盐存储。
- 敏感信息（证件号）加密存储/脱敏展示。
- 全链路 HTTPS。

## 12. 可观测性

### 12.1 统一日志规范
- JSON 结构化日志，含 `traceId`/`spanId`/`userId`/`service`。
- 日志级别：ERROR（告警）/ WARN / INFO（关键链路）/ DEBUG（开发）。
- 敏感信息脱敏（手机号、证件号）。

### 12.2 链路追踪
- 全链路 `traceId` 由网关生成，透传至所有服务（MQ 消息也携带）。
- 基于 Micrometer Tracing（Spring Boot 4.0 替代 Sleuth）。

### 12.3 监控指标
- 业务：下单 QPS、下单成功率、库存预扣失败率、支付成功率、订单超时取消率。
- 系统：CPU/内存/GC、DB 连接池、Redis 命中率、Kafka 积压。
- 告警：超卖（零容忍，立即告警）、核心接口错误率 > 阈值、MQ 积压超限。

## 13. 部署架构与环境规划

### 13.1 环境划分
| 环境 | 用途 | Nacos namespace |
|---|---|---|
| dev | 本地开发 | dev |
| test | 联调测试 | test |
| prod | 生产 | prod |

### 13.2 容器化思路
- 各服务打 Docker 镜像，按服务独立部署、独立扩缩容。
- 抢票前对 order/program 服务**水平扩容**（HPA）。
- 中间件（Redis/Kafka/ES/MySQL）独立集群，按需扩容。
- 抢票场景下，Redis 与 order 服务就近部署（同机房/可用区），降低预扣延迟。

### 13.3 容量规划（示例）
- order 服务：单实例 5000 QPS，抢票期扩至 10 实例 → 5 万 QPS。
- Redis：主从 + 哨兵/集群，库存 Key 分散到不同分片避免热点单点。
- Kafka：按预期吞吐设置分区数，消费组并发消费。

## 14. 风险清单与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| **Boot4+Cloud2025+Nacos 集成问题（#4183）** | 启动/配置加载异常 | 锁精确小版本；启用本地快照兜底；跟踪官方修复（§3.2） |
| 热点节目库存 Key 单点 | 单 Redis 节点打满 | 库存 Key 分片；Redis 集群；极端时按票档拆分 |
| 消息积压 | 超时取消/支付通知延迟 | 监控积压告警；动态扩消费实例；DLQ 兜底 |
| 最终一致延迟 | DB 与 ES/库存短暂不一致 | 接受读最终一致；定时对账；关键交易走强一致链路 |
| 超卖 | 业务事故 | 三级防超卖（§9.2）；超卖零容忍告警 |
| 缓存与 DB 不一致 | 用户看到过期库存 | 库存实时读 Redis（预扣后值）；节目信息接受短延迟 |
| Seata 长事务 | 全局锁占用、性能下降 | 关键写才纳入事务；耗时操作移出（§8.3.2） |
| 抢票前扩容不足 | 服务雪崩 | 压测预估容量；提前扩容；限流兜底保护 |
