## 2. 项目背景与目标

### 2.1 背景
在线票务（演出、体育、亲子等）是典型的**读多写少、瞬时写并发极高**的 C 端交易场景。其核心矛盾并非 CRUD 本身，而是：

- **抢票尖峰**：热门节目开售瞬间，QPS 可在数秒内从日常百级飙升至十万级，对系统吞吐、库存一致性与可用性构成三重挑战。
- **超卖零容忍**：一张票不能卖给两个人，库存一致性是底线。
- **链路敏感**：用户对下单响应、支付体验极其敏感，核心链路必须保持低延迟。

「大麦」项目作为**个人技术展示项目**，以"高并发抢票"为切入点，系统性落地分布式架构、缓存与一致性、消息异步化、分库分表等核心技术能力。

### 2.2 目标
| 维度 | 目标 |
|---|---|
| 功能完整性 | 覆盖用户登录、演出浏览、选座购票、订单支付与取消全流程 |
| 性能 | 抢票核心链路 P99 < 500ms，单集群目标支撑 ≥ 5 万 QPS 抢票 |
| 一致性 | 库存零超卖（强一致）；订单/支付最终一致 |
| 可用性 | 核心服务可用性 ≥ 99.9%（年宕机 < 8.76h） |
| 技术展示 | 形成可直接用于面试/述职的项目亮点集（Lua 预扣、布隆过滤、Seata 长事务处理等） |

### 2.3 非目标（Non-Goals）
- 不做真实第三方支付通道对接（支付为模拟实现）。
- 不做商家入驻/票务上架的完整后台 PaaS（仅提供基础运营管理能力）。
- 不做客户端 App/H5 完整前端工程（仅约定接口契约，便于联调）。
- 不做生产级风控（仅做基础防刷限流示意）。

## 3. 项目范围与边界

### 3.1 纳入范围（In Scope）
- 用户域：注册、登录（JWT）、认证鉴权、个人信息维护。
- 演出节目域：节目浏览、分类筛选、智能搜索（ES）、节目详情、场次座位图。
- 订单域：选座、预下单（锁座）、正式下单、订单查询、主动取消。
- 支付域：发起支付（模拟）、支付结果回调、退款（模拟）。
- 基础数据域：节目类型、地区、轮播图、字典数据。
- 平台支撑：网关、注册配置中心、限流熔断、分布式事务、消息异步化、分库分表。

### 3.2 排除范围（Out of Scope）
- 真实支付通道集成（微信/支付宝商户对接）。
- 复杂营销（拼团、秒杀券、分销）。
- 完整 B 端运营后台与审核工作流。
- 物流/纸质票寄送。

## 4. 名词术语表

| 术语 | 英文/缩写 | 含义 |
|---|---|---|
| 节目 | Program | 一场可售卖的演出/比赛实体，如"周杰伦 嘉年华世界巡演 - 北京站" |
| 场次 | Show | 节目下的具体时间场次，如"2026-07-01 19:30 场"，一个节目可有多场次 |
| 票档 | TicketType | 场次下的票价档位，如"VIP 1880 / 980 / 580 / 380" |
| 座位 | Seat | 物理座位（带座号），属于某场次的座位图 |
| 库存 | Stock | 某（场次+票档）或（场次+座位）的可售数量 |
| 订单 | Order | 一次购票交易记录 |
| 抢票 | Flash Sale | 热门节目开售瞬间的高并发下单行为 |
| 预扣 | Pre-deduct | 下单瞬间先在 Redis 扣减库存，落单后再异步同步 DB |
| 幂等令牌 | Idempotent Token | 防重复提交的一次性票据 |
| 幂等性 | Idempotency | 同一请求重复执行结果一致 |
| 超卖 | Over-sell | 库存为 N 却卖出 > N 张票 |
| 缓存穿透 | Cache Penetration | 查询不存在的数据，每次穿透到 DB |
| 缓存击穿 | Cache Breakdown | 热点 Key 失效瞬间，海量请求同时打 DB |
| 缓存雪崩 | Cache Avalanche | 大量 Key 同时失效，请求涌入 DB |
| 延迟队列 | Delay Queue | 消息延迟投递，用于订单超时取消 |

## 7. 功能需求详述

> 本节按服务域拆分到子功能点，给出**输入 / 处理 / 输出 / 关键规则**四要素。

### 7.1 用户服务（damai-user）

#### 7.1.1 注册（UC-U-01）
- **输入**：手机号、密码、短信验证码。
- **处理**：校验验证码 → 唯一性校验（Redis 布隆过滤器 + DB）→ 密码加密（BCrypt）→ 落库。
- **输出**：用户 ID、JWT。
- **关键规则**：手机号唯一；验证码 5 分钟有效；同号 60s 内不可重复获取验证码（防刷限流）。

#### 7.1.2 登录（UC-U-02）
- **输入**：手机号 + 密码（或验证码）。
- **处理**：账号校验 → 密码比对 → 签发 JWT（含 userId、过期时间）→ RefreshToken 存 Redis。
- **输出**：accessToken、refreshToken、用户基本信息。
- **关键规则**：登录失败 5 次锁定 15 分钟；JWT 无状态 + RefreshToken 可吊销。

#### 7.1.3 认证鉴权
- 网关统一解析 JWT，将 `userId` 注入请求头透传至下游服务。
- 下游服务按角色/资源做细粒度校验。

#### 7.1.4 个人信息 / 观演人（UC-U-04 / UC-U-05）
- 维护昵称、头像、收货地址；维护观演人（姓名 + 证件号），用于实名购票校验。

### 7.2 演出节目服务（damai-program）

#### 7.2.1 首页推荐 / 轮播（UC-P-01）
- **输入**：城市、频道（演唱会/话剧/…）。
- **处理**：命中 Redis 缓存则直接返回；否则回源并异步刷新。
- **输出**：轮播图列表 + 推荐节目分页列表。
- **关键规则**：首页数据 CDN 思路（边缘缓存 + 短 TTL），允许最终一致。

#### 7.2.2 分类浏览（UC-P-02）
- 按节目类型、地区、时间、价格区间组合筛选。
- 数据源：ElasticSearch（支撑复杂多维筛选 + 排序）。

#### 7.2.3 智能搜索（UC-P-03）
- **输入**：关键词（节目名/艺人/场馆）。
- **处理**：分词检索 ES，支持纠错、高亮、按相关度/热度/时间排序。
- **输出**：搜索结果分页。
- **关键规则**：搜索走 ES，MySQL 仅作为权威写库，二者通过**异步消息 + 定时校对**同步（详见技术文档 §7.3）。

#### 7.2.4 节目详情（UC-P-04）
- 聚合：节目基础信息 + 场次列表 + 各票档及**实时库存**。
- **关键规则**：库存读 Redis（预扣后实时值），节目信息读缓存；二者分离避免热点耦合。

#### 7.2.5 场次座位图（UC-P-05）
- 本期支持两种售卖模式：
  - **票档级（先到先得）**：用户选票档与数量，系统自动分配座位（实现简单，抢票主流模式）。
  - **座位级（可选增强）**：用户在座位图上自选座位，提交时锁定所选座位。
- 本期默认实现**票档级**，座位级作为亮点增强项标注。

#### 7.2.6 节目发布 / 库存预热（UC-P-06）
- 运营发布节目/场次/票档后，触发**库存预热**：DB 库存 → 写入 Redis（含 Lua 原子结构）、节目信息写入 ES、节目 ID 写入布隆过滤器。

### 7.3 订单服务（damai-order）

#### 7.3.1 选座 / 选票档（UC-O-01）
- 用户选定场次 + 票档 + 数量，进入下单流程；前端领取**幂等令牌**（防重复提交）。

#### 7.3.2 预下单 / 下单（UC-O-02 / UC-O-03）⭐核心
- **输入**：场次ID、票档ID、数量、幂等令牌、观演人信息。
- **处理**（核心链路，详见 §8.1）：
  1. 幂等校验（令牌一次性消费）。
  2. **Redis Lua 预扣库存**（原子，防超卖）。
  3. 落订单（待支付状态，写 DB + 分库分表）。
  4. 投递"订单超时取消"延迟消息（15 分钟）。
  5. 异步通过 Kafka 通知库存服务做 DB 库存最终扣减与对账。
- **输出**：订单号、待支付金额、支付截止时间。
- **关键规则**：
  - 幂等：同一令牌只成功一次。
  - 防超卖：Lua 原子扣减，库存不足直接失败。
  - 锁库存时长：15 分钟，超时未支付自动取消并归还。

#### 7.3.3 订单查询（UC-O-04）
- 我的订单：按状态（待支付/已支付/已取消/已退款）筛选，分页。
- 订单详情：含场次、票档、数量、金额、状态、支付信息。

#### 7.3.4 主动取消（UC-O-05）
- 仅"待支付"订单可取消；取消后**归还 Redis 库存** + 异步归还 DB 库存 + 更新订单状态。

#### 7.3.5 超时自动取消（UC-O-06）
- 由 Kafka 延迟消息触发：订单超时未支付 → 置为"已取消(超时)" → 归还库存 → 通知用户。

### 7.4 支付服务（damai-pay）

#### 7.4.1 发起支付（UC-PAY-01）
- **输入**：订单号、支付方式（模拟）。
- **处理**：校验订单状态（待支付）→ 生成支付单 → 返回模拟支付参数。
- **关键规则**：支付单与订单一一对应；支付单有效期 = 订单剩余有效时间。

#### 7.4.2 支付结果回调（UC-PAY-02）⭐核心
- **输入**：订单号、支付结果（成功/失败）、回调签名。
- **处理**：
  1. **幂等校验**（回调可能重复投递）。
  2. 更新支付单状态。
  3. 通过 Kafka 异步通知订单服务更新订单为"已支付"。
  4. 触发出票（模拟）。
- **关键规则**：回调幂等是底线；订单状态机不可逆（已支付不可回退为待支付）。

#### 7.4.3 支付状态查询 / 对账（UC-PAY-03）
- 定时任务扫描"已发起未回调"的支付单，主动查询支付方（模拟），补偿状态。

#### 7.4.4 退款（UC-PAY-04）
- 模拟流程：发起退款 → 审批 → 退款成功回调 → 订单置"已退款"。

### 7.5 基础数据服务（damai-base）
- 节目类型、地区、轮播图、字典的 CRUD，作为被依赖方提供基础数据。
- 数据量小、读多写少，全量入 Redis 缓存。

## 8. 核心业务流程

### 8.1 抢票下单全链路（最核心）⭐

```
用户                 网关            订单服务         Redis          Kafka          库存/DB
 │                    │                 │               │              │              │
 │─ 1.领幂等令牌──────▶│────────────────▶│               │              │              │
 │◀─── 令牌 ──────────│◀────────────────│               │              │              │
 │                    │                 │               │              │              │
 │─ 2.提交下单────────▶│────────────────▶│               │              │              │
 │   (令牌+场次+票档+数量)              │               │              │              │
 │                    │                 │─3.幂等校验(消费令牌,Redis)─▶│              │
 │                    │                 │◀── ok ───────────────────────│              │
 │                    │                 │─4.Lua 预扣库存─────────────▶│              │
 │                    │                 │◀── 扣减结果(原子)───────────│              │
 │                    │                 │  (库存不足→直接失败,快速返回) │              │
 │                    │                 │─5.落订单(待支付,DB分片)───────────────────▶│
 │                    │                 │─6.投递超时取消延迟消息────────▶│              │
 │                    │                 │─7.异步通知DB库存扣减──────────▶│─────────────▶│
 │◀─── 8.订单号+待支付金额+截止时间────│◀──────────────────────────────│              │
 │                    │                 │               │              │              │
 │   (此时用户拿到订单,15分钟内完成支付) │               │              │              │
```

**设计要点**：
1. **幂等在前**：令牌一次性消费，挡住 99% 的重复提交。
2. **Redis 预扣优先**：核心链路只触碰 Redis（Lua 原子），DB 写在链路后段，保证 P99 低延迟。
3. **DB 库存异步同步**：通过 Kafka 解耦，核心链路不等 DB。
4. **延迟消息兜底**：超时未支付自动归还，杜绝僵尸库存。
5. **失败快速返回**：库存不足时 Lua 立即返回失败，不占用下游资源。

### 8.2 支付成功回调流程

```
支付方(模拟)        支付服务            Kafka           订单服务         出票(模拟)
 │                    │                  │                │                │
 │─ 支付结果回调──────▶│                  │                │                │
 │                    │─ 幂等校验(Redis)─▶│                │                │
 │                    │─ 更新支付单(DB)──▶│                │                │
 │                    │─ 异步通知订单已付─▶│                │                │
 │                    │                  │─ 消费 ─────────▶│                │
 │                    │                  │                │─ 状态机校验     │
 │                    │                  │                │─ 更新订单已付   │
 │                    │                  │                │─ 触发出票──────▶│
 │◀── 回调 ACK ───────│                  │                │                │
```

**设计要点**：回调幂等（Redis 防重 + 状态机校验双重保障）；支付与订单通过 Kafka 解耦，互不阻塞。

### 8.3 订单超时自动取消 + 归还库存

```
订单服务(下单时)        Kafka(延迟15min)        订单服务(消费)          Redis库存         DB库存
 │                        │                        │                       │                │
 │─ 投递延迟消息──────────▶│                        │                       │                │
 │                        │  (15分钟后投递)         │                       │                │
 │                        │─ 延迟消息──────────────▶│                       │                │
 │                        │                        │─ 查订单当前状态         │                │
 │                        │                        │  (已支付→直接丢弃)      │                │
 │                        │                        │  (待支付→继续取消)      │                │
 │                        │                        │─ 更新订单为已取消(超时)  │                │
 │                        │                        │─ 归还 Redis 库存───────▶│                │
 │                        │                        │─ 异步归还 DB 库存───────────────────────▶│
```

**设计要点**：消费时**二次校验订单状态**（延迟窗口内可能已支付），避免误归还；库存归还同样 Redis 同步 + DB 异步。

## 9. 非功能性需求（NFR）

| 维度 | 指标 | 说明 |
|---|---|---|
| 并发 | 抢票核心链路单集群 ≥ 5 万 QPS | 通过 Lua 预扣 + 异步化达成 |
| 响应时间 | 核心链路 P99 < 500ms；下单 P99 < 300ms | 不含第三方支付耗时 |
| 可用性 | 核心服务 ≥ 99.9% | 网关/订单/支付为主 |
| 一致性 | 库存强一致（零超卖）；订单/支付最终一致 | 详见专项需求 §10 |
| 容量 | 单节目抢票支撑 ≥ 100 万人次同时参与 | 通过分片 + 限流兜底 |
| 安全 | 接口防刷、JWT 鉴权、敏感信息加密 | 详见安全设计 |
| 可观测 | 全链路 traceId、关键指标监控、告警 | 详见技术文档 §12 |
| 数据持久 | 订单数据分库分表，可水平扩展 | ShardingSphere |

## 10. 高并发与一致性专项需求 ⭐

> 本节是项目的**核心亮点**，也是评审/面试最关注的章节。

### 10.1 库存防超卖（三级保障）
| 层级 | 手段 | 作用 |
|---|---|---|
| L1 | **Redis Lua 预扣**（原子脚本） | 核心链路兜底，挡住绝大多数并发 |
| L2 | **Redisson 分布式锁** | 热点票档的细粒度串行化（极端竞争场景） |
| L3 | **DB 乐观锁/行锁兜底**（`stock - ? >= 0` 条件更新） | 数据最终正确性底线 |

> 设计原则：**越靠前的拦截越廉价**，L1 挡住 99.9% 流量，L2/L3 作为安全网。

### 10.2 防缓存穿透
- **布隆过滤器**：节目发布时将节目 ID 写入布隆过滤器；查询前先过布隆，不存在的 Key 直接拒绝，不打 DB。

### 10.3 防缓存击穿
- 热点节目详情采用**逻辑过期 + 互斥重建**：逻辑过期时间到达时，由第一个线程异步重建，其余线程返回旧值，避免瞬间击穿。

### 10.4 防缓存雪崩
- TTL 加随机抖动（`base + random`）；多级缓存（本地 + Redis）；关键数据预热。

### 10.5 幂等性
- 下单、支付、回调均需幂等：
  - **下单**：幂等令牌（前端领、后端一次性消费）。
  - **支付回调**：订单号 + 状态机双重幂等。
  - **重复提交**：Redis 防重锁（key = `userId:action:bizId`，短 TTL）。

### 10.6 分布式事务（跨服务一致性）
- 下单扣库存跨"订单服务 + 库存（节目服务）"：以 **Seata AT** 为最终兜底；同时通过 Kafka 异步对账做最终一致。
- **长事务规避**：核心链路不放 Seata 全局事务（避免 RM 长时间占用），仅关键写操作纳入；耗时操作（通知、出票）移到事务外异步。

### 10.7 限流熔断
- **网关层**：全局 QPS 限流、IP/用户维度防刷。
- **服务层**：Sentinel 接口限流 + **热点参数限流**（按节目ID/票档ID 限流，防热点打垮）。
- **熔断**：下游依赖（ES/DB）异常时快速失败，避免级联雪崩。

## 11. 接口规范

### 11.1 统一返回结构
所有接口统一返回 `ApiResult<T>`：
```
ApiResult<T> {
  code:    string   // 业务码，见 §11.2
  message: string   // 提示信息
  data:    T        // 业务数据
  traceId: string   // 链路追踪 ID
  success: boolean  // 是否成功（code == SUCCESS）
}
```

### 11.2 错误码体系（分级段位）
| 段位 | 含义 | 示例 |
|---|---|---|
| `SUCCESS` / `0` | 成功 | — |
| `1xxxx` | 通用错误（参数/权限/系统） | `10001` 参数非法 |
| `2xxxx` | 用户域错误 | `20001` 手机号已注册 |
| `3xxxx` | 节目域错误 | `30001` 节目不存在/已下架 |
| `4xxxx` | 订单域错误 | `40001` 库存不足；`40002` 重复提交 |
| `5xxxx` | 支付域错误 | `50001` 订单状态非法 |
| `9xxxx` | 第三方/基础设施错误 | `90001` 支付通道异常 |

> 原则：**业务可预知错误返回明确业务码 + 中文提示**；**不可预知异常由全局异常处理器兜底**（`@RestControllerAdvice`），返回统一 `99999` 系统异常并记录 traceId。

### 11.3 RESTful 约定（概要）
- 资源化路径：`GET /program/{id}`、`POST /order`。
- 分页：`pageNum` / `pageSize`，返回 `total` + `list`。
- 时间：统一 ISO-8601 / 时间戳，UTC 存储、展示按城市时区。
