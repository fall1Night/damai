## 三、关键设计决策

### 3.1 库存预扣 Lua 脚本设计（⭐核心）

**设计**：
- 使用 Redis Lua 脚本实现原子操作
- 预扣脚本：`GET → 判断 >= n → DECRBY`，返回 1（成功）/ -1（库存不足）/ -2（Key 不存在）
- 归还脚本：`GET → INCRBY`，返回 1（成功）/ -2（Key 不存在）

**为什么必须 Lua**：
- 普通 `GET → DECRBY` 两步操作之间存在时间窗
- 并发下多个线程都读到 `stock >= n`，全部扣减导致超卖
- Lua 脚本在 Redis 中单线程原子执行，杜绝竞态

**调用方式**：
```java
// 通过 Redisson RScript.evalSha() 执行
Long result = redissonClient.getScript().evalSha(
    RScript.Mode.READ_WRITE,
    deductScriptSha,
    RScript.ReturnType.INTEGER,
    Collections.singletonList(stockKey),
    quantity
);
```

### 3.2 三级防超卖联动

| 层级 | 实现 | 触发条件 | 性能 |
|---|---|---|---|
| L1 | Redis Lua 预扣（StockManager） | 每次下单 | 极高（核心拦截层） |
| L2 | Redisson 分布式锁（DistributedLockHelper） | 仅极端竞争/座位级售卖时 | 中（牺牲并发换强串行） |
| L3 | DB 条件更新（TicketTypeManager.updateStock） | 异步落 DB | 低（最终正确性底线） |

### 3.3 库存预热流程

```
1. 查询节目信息（DB）
2. 查询所有场次（DB）
3. 遍历场次，查询每个票档（DB）
4. 初始化 Redis 库存 Key（stock:{showId}:{ticketTypeId} = saleStock）
5. 节目 ID 写入布隆过滤器
6. 发送 Kafka 消息，异步同步到 ES
```

### 3.4 ES 搜索设计

**分类浏览**：
- 使用 BoolQuery 多条件筛选（typeCode、cityCode、status）
- 按热度排序（heat DESC）
- 分页查询

**智能搜索**：
- 使用 multi_match 查询（title、artist、venue、description）
- 支持 ik 分词
- 只查询在售状态

**异步同步**：
- 节目变更 → 发 Kafka 消息 → 消费者更新 ES
- 定时校对任务（每小时）兜底消息丢失

### 3.5 缓存策略

| 数据 | Key 格式 | TTL | 说明 |
|---|---|---|---|
| 节目详情 | `program:detail:{programId}` | 30 分钟 | 按 ID 查询 |
| 场次列表 | `program:shows:{programId}` | 30 分钟 | 按节目查询 |
| 票档列表 | `program:ticket-types:{showId}` | 30 分钟 | 按场次查询 |
| 库存 | `stock:{showId}:{ticketTypeId}` | 与开售周期一致 | 预扣核心 |
| 空值占位 | 同上 | 60 秒 | 防穿透 |

### 3.6 布隆过滤器防穿透

**设计**：
- 节目存在性校验使用布隆过滤器预判
- 预估元素数量：100 万
- 误判率：0.01%

**流程**：
1. 节目发布时添加到布隆过滤器
2. 查询详情时先过布隆过滤器
3. 布隆判断"可能存在"再查缓存/DB

## 九、总结

M2 节目服务（步骤 10-16）已成功完成，共生成 **22 个文件**，约 **1790 行代码**。实现了节目浏览、搜索、详情、库存预热与 Lua 预扣等核心功能，特别是库存预扣模块是整个抢票系统的核心。

关键设计亮点：
1. **Lua 原子预扣**：Redis Lua 脚本实现"读取-判断-扣减"原子操作，杜绝超卖
2. **三级防超卖**：L1 Redis Lua → L2 分布式锁 → L3 DB 乐观锁，层层兜底
3. **库存预热**：DB→Redis 初始化 + ES 同步 + 布隆过滤器写入
4. **异步同步**：Kafka 消息 + 定时校对，保证 ES 与 MySQL 最终一致
5. **布隆过滤器**：节目存在性预判，减少缓存穿透

下一步将继续执行 M3 订单服务（步骤 17-22），完成抢票下单全链路、幂等、延迟取消、分库分表等核心功能。
