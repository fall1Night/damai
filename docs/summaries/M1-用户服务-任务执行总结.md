## 三、关键设计决策

### 3.1 JWT 双 Token 机制

**设计**：
- `AccessToken`：有效期 2 小时，无状态，用于接口鉴权
- `RefreshToken`：有效期 7 天，存 Redis，支持主动吊销

**优势**：
- AccessToken 短有效期，降低泄露风险
- RefreshToken 存 Redis，支持登出/改密时主动吊销
- 用户无感刷新，体验流畅

### 3.2 登录安全策略

**设计**：
- 登录失败 5 次锁定 15 分钟
- 使用 Redis INCR + TTL 实现，无需额外定时任务
- 登录成功后清除失败计数

**实现**：
```java
// Key: user:login:fail:{mobile}
// TTL: 15 分钟
// 阈值: 5 次
```

### 3.3 布隆过滤器防穿透

**设计**：
- 手机号唯一性校验使用布隆过滤器预判
- 预估元素数量：1000 万
- 误判率：0.01%

**流程**：
1. 注册时先过布隆过滤器
2. 布隆判断"可能存在"再查 DB
3. 注册成功后添加到布隆过滤器

### 3.4 缓存策略

| 数据 | Key 格式 | TTL | 说明 |
|---|---|---|---|
| 用户信息 | `user:info:{userId}` | 30 分钟 | 按 ID 查询 |
| 用户信息 | `user:info:mobile:{mobile}` | 30 分钟 | 按手机号查询 |
| 观演人列表 | `user:viewers:{userId}` | 30 分钟 | 按用户查询 |
| 空值占位 | 同上 | 60 秒 | 防穿透 |
| 登录失败计数 | `user:login:fail:{mobile}` | 15 分钟 | 登录锁定 |
| RefreshToken | `user:refresh:{userId}` | 7 天 | Token 吊销 |

### 3.5 四层架构遵循

```
Controller（参数校验、调用 Service、组装返回）
    ↓
Service（业务编排、事务边界、调用 Manager）
    ↓
Manager（缓存/DB/中间件组合操作）
    ↓
DAO（数据访问，MyBatis-Plus Mapper）
```

**约束**：
- `@Transactional` 只在 Service 层
- Service 不直接操作 Redis，通过 Manager 封装
- Controller 不直接调用 DAO

### 3.6 内部接口隔离

**设计**：
- 用户接口：`/user/**`（对外暴露，网关路由）
- 内部接口：`/user/inner/**`（仅供服务间 Feign 调用，网关不暴露）

**实现**：
- `GET /user/inner/getById`：供其他服务查询用户信息
- `GET /user/inner/viewer/list`：供其他服务查询观演人列表

## 四、ErrorCode 枚举更新

### 4.1 新增错误码

| 错误码 | 枚举名 | 说明 |
|---|---|---|
| `20001` | `USER_ALREADY_EXISTS` | 手机号已注册 |
| `20006` | `USER_LOGIN_LOCKED` | 登录失败次数过多，账号已被锁定 15 分钟 |
| `20007` | `SMS_CODE_EXPIRED` | 验证码已过期 |
| `20008` | `SMS_CODE_ERROR` | 验证码错误 |
| `20010` | `TOKEN_INVALID` | Token 无效 |
| `20012` | `VIEWER_LIMIT_EXCEEDED` | 观演人数量已达上限（最多 5 个） |
| `20013` | `VIEWER_NOT_FOUND` | 观演人不存在 |

### 4.2 命名规范

- 用户域错误码统一使用 `USER_` 前缀
- 观演人域错误码统一使用 `VIEWER_` 前缀
- 短信验证码错误码统一使用 `SMS_CODE_` 前缀
- Token 错误码统一使用 `TOKEN_` 前缀
